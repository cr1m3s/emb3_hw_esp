## 1. Схема підключення
- Кнопка → GPIO мікроконтролера
- Та ж кнопка → канал логічного аналізатора
- Підтягування — внутрішнє або резистор

## 2. Хід роботи
- Натиснути кнопку 10 разів із різною швидкістю й силою.
- Логічним аналізатором записати форму сигналу кожного натискання.
- Для кожного натискання визначити:
    - скільки імпульсів брязкоту виникло насправді
    - тривалість брязкоту (у мілісекундах)

## 3. Порівняти:
- кількість імпульсів, які бачить логічний аналізатор
- кількість імпульсів, які зареєстрував мікроконтролер

## 4. Питання для аналізу
- Чи співпадає кількість імпульсів?
``` 
Не співпадає:
Bounce: 12      Debounced: 10   Ratio: 1.20x
```
- У яких випадках мікроконтролер «пропускає» частину брязкоту, а коли навпаки — «перерахує зайве»?
```
Пропускає - недостатньо сильний рівень сигналу.
Перераховує - випадкове підвищення напруги.
```

- Яка середня тривалість брязкоту для вашої кнопки?
```
Більшість сигналів в логічному аналізаторі мають тривалість більшу за 100 мс. Чітко сказати тривалість брязкоту сказати не можу, але він менший за 100мс.
```
- Який мінімальний debounce (затримка / логіка фільтрації) був би достатній?
```
Достатньо було 100 мс.
```
Не впевнений в достовірності цих данних, але дуже схоже на правду:

| Тип кнопки          |Час дебаунсу |
|---------------------|-------------|
| Якісна тактова      |20-50 мс     |
| Звичайна кнопка     |50-100 мс    |
| Погана кнопка       |100-150 мс   |
| Механічний перемикач|10-20 мс     |

4. Додаткове (опційно)

    Запустити другий варіант коду з програмним debounce (наприклад, 5–10 мс) і переконатися, що кількість імпульсів на виході стає рівно 1.
    Додати резистор та конденсатор і переконатися у відсутності брязкоту.
